## Day 9 - 4 June - Schemas + Provider Interface

### Guide Objective

Define the structured data contract for the RCA engine and create a provider boundary.

### Work Completed

Implemented:

```text
schemas.py
providers/base.py
```

### `schemas.py`

Created `RCAInput`, which accepts:

- `problem_statement`
- optional `context`

Validation behavior:

- trims whitespace;
- rejects blank text;
- rejects problem statements shorter than 10 characters;
- forbids unexpected extra fields.

Created `WhyEntry`, which represents one step in the 5 Whys chain:

- `index`: integer from 1 to 5;
- `question`: why question;
- `answer`: causal answer that deepens the chain.

Created `RCAReport`, which represents complete model output:

- `problem`
- `summary`
- exactly five `why_chain` entries;
- `root_cause`
- `contributing_factors`
- `recommendations`
- `confidence`
- optional provider metadata: `source_model`, `prompt_version`, `latency_seconds`

Important validation:

```text
why_chain must contain exactly five entries.
why_chain indexes must be exactly [1, 2, 3, 4, 5].
contributing_factors and recommendations cannot contain blank items.
```

### `providers/base.py`

Created abstract provider interface:

```text
RCAProvider
```

Required members:

- `model` property;
- `generate(rca_input, prompt_version, strict_retry=False) -> RCAReport`.

This lets the rest of the app call one stable interface without caring whether the underlying model is local Ollama, hosted-open, or another future backend.

The provider:

1. Reads settings from `config.py`.
2. Creates an OpenAI-compatible client pointed at Ollama:

```text

```

```text
instructor.from_openai(..., mode=instructor.Mode.JSON)
```

4. Calls the local model with:


5. Adds metadata after generation:

- `source_model`
- `prompt_version`
- `latency_seconds`

## Day 11 - 6 June - Agent Orchestration + Prompt V1

### Guide Objective

Create versioned prompts and a core orchestration function that chooses the provider, builds prompts, and returns a validated RCA report.

### Work Completed

Implemented:

```text
prompts.py
config.py
rca_agent.py
```

### `prompts.py`

Created two prompt versions:

```text
v1
v2
```

`v1` is the basic RCA analyst prompt:

- clear role;
- JSON schema instruction;
- concise, blameless RCA output.

`v2` is stricter:

- no markdown fences;
- exactly five indexed why entries;
- whys must deepen from symptom to mechanism to process/system cause;
- root cause must not simply restate the symptom;
- recommendations must directly address the root cause;
- confidence must reflect available evidence.

Created:

```text
build_messages(...)
```

This returns chat messages for the selected prompt version and can add a stricter retry instruction.

### `config.py`

Created `Settings` dataclass and `get_settings()` helper.

Settings read from `.env`:

- `LLM_PROVIDER`
- `OLLAMA_BASE_URL`
- `RCA_MODEL`
- `RCA_PROMPT_VERSION`
- `RCA_MAX_RETRIES`
- `RCA_EVAL_MODELS`
- `OUTPUT_DIR`

### `rca_agent.py`

Created:

```text
build_provider(settings=None)
generate_rca(problem, context=None, provider=None, settings=None)
```

`generate_rca(...)` does the main orchestration:

1. Builds `RCAInput`.
2. Chooses provider from settings.
3. Calls provider with configured prompt version.
4. If generation fails, retries once with stricter prompt instructions.

## Day 12 - 7 June - Model Eval

### Guide Objective

Create a small rubric and eval runner that compares 2-3 local models across four incidents.

### Work Completed

Implemented:

```text
eval/rubric.md
eval/run_eval.py
eval/results.md
eval/results.json
```

### `eval/rubric.md`

Defined a 10-point rubric:

| Criterion | Points |
| --- | ---: |
| Valid schema | 2 |
| Five distinct whys | 2 |
| Deepening causal chain | 2 |
| Root cause quality | 2 |
| Recommendations | 1 |
| Latency | 1 |

Incident set:

1. Login API returns HTTP 500 immediately after a deployment.
2. Checkout requests time out after a database migration.
3. Background invoice jobs stopped running after a scheduler change.
4. CPU usage spikes after enabling a new analytics endpoint.

### `eval/run_eval.py`

The eval runner:

1. Loads candidate models from `RCA_EVAL_MODELS`.
2. Runs each incident through the provider layer.
3. Validates every response as `RCAReport`.
4. Scores each output with the rubric.
5. Writes Markdown results to `eval/results.md`.
6. Writes raw result rows to `eval/results.json`.

The script includes repo-root import bootstrapping so it works when run as:

```powershell
python eval\run_eval.py
```

### Eval Result

The actual run compared:

```text
qwen2.5:7b
llama3.2:latest
```

Summary:

| Model | Avg Score | Avg Latency | Valid Runs | Decision |
| --- | ---: | ---: | ---: | --- |
| `qwen2.5:7b` | 9.50/10 | 59.76s | 4/4 | selected |
| `llama3.2:latest` | 9.62/10 | 78.05s | 4/4 | comparison |

Although `llama3.2:latest` scored 0.12 higher, `qwen2.5:7b` was selected because the score gap was tiny and Qwen had better average latency and consistency.
